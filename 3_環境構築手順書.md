# 3. 環境構築手順書（開発開始〜教授編集〜成果物生成まで）

## 3-1. 目的
本書は、採用技術スタック（静的サイト + ブラウザ編集CMS + CI + 観測）に基づき、
- あなたが開発を開始できる
- 教授がブラウザで更新できる
- CIでビルド・品質チェック・成果物ZIP生成ができる
状態までの手順を定義する。

## 3-2. 前提
- 公開は慶應Webサーバ `public_html`（静的配置）
- GitHubを利用可能
- 教授はGitHubアカウントを作成・利用可能
- デプロイ方式は未決（当面：成果物ZIPをあなたがCyberduckで反映）

---

## 3-3. あなたのローカル環境（開発PC）準備
### 3-3-1. 必要ソフト
- Node.js（LTS推奨）
- Git
- エディタ（VS Code等）

### 3-3-2. リポジトリ作成
1) GitHubで新規リポジトリ作成（Private推奨）
2) mainブランチ保護
- PR必須
- CI成功必須
3) 教授をCollaboratorとして追加（Write権限）
4) 2FAを有効化（教授・あなた）

---

## 3-4. 静的サイトプロジェクト作成（例：Astro）
### 3-4-1. 初期化
- 新規プロジェクトを作成
- ローカル起動してトップページが表示されることを確認

### 3-4-2. 基本設定
- TypeScript（必要なら）
- フォーマッタ/リンタ（必要範囲）
- 画像/ファイル命名規則（全角禁止、短い英数、スペース禁止）をREADMEに明記

---

## 3-5. コンテンツ格納（SSOT）
### 3-5-1. 推奨配置
- content/pages：固定ページ原稿
- content/news：お知らせ
- content/publications：業績データ

### 3-5-2. 最小テンプレ
- Top
- Research
- Publications

- Contact
- News

## 3-6. CMS（Decap CMS）導入（教授のブラウザ編集を実現）
### 3-6-1. 管理画面を用意
- admin/index.html を追加
- admin/config.yml を追加

### 3-6-2. コンテンツモデル（最小で開始）
- Pages
  - title
  - slug
  - body
- News
  - title
  - date
  - body
  - tags（任意）
- Publications
  - year
  - authors
  - title
  - venue（誌/会議）
  - category
  - doi_or_url（任意）

### 3-6-3. GitHub OAuth設定（必須）
1) GitHubでOAuth App作成
- Homepage URL：プレビュー環境のURL
- Authorization callback URL：CMSのコールバックURL
2) CMS設定（config.yml）に反映

### 3-6-4. 教授の操作テスト
- 教授が/adminへアクセスしてログインできる
- Newsを1件追加できる
- PR（またはブランチ）が作られる

---

## 3-7. CI（GitHub Actions）構築

### 3-7-1. PRチェック（pr-check.yml）
PR作成/更新のたびに以下を実行：
- 依存関係インストール
- ビルド
- リンクチェック
- 全角ファイル名検出
- 画像サイズ上限チェック（任意）
- Lighthouse（任意、重ければ後回し）

### 3-7-2. 成果物生成（build-release.yml）
mainマージのたびに以下を実行：
- ビルド
- distをZIP化
- ArtifactsまたはReleaseに添付（ロールバック用）

---

## 3-8. 観測（GA4 / Sentry）導入

### 3-8-1. GA4
1) GA4プロパティ作成
2) 測定IDを取得
3) サイトに計測タグを組み込み
4) KPIイベントを実装
- contact_click
- joinus_view
- doi_click
- scholar_click
- pdf_click（許諾OKのみ）
5) リアルタイムでイベント確認

### 3-8-2. Sentry
1) プロジェクト作成
2) DSN取得
3) フロントに導入
4) 通知設定（あなた宛）

---

## 3-9. プレビュー環境（推奨：PRごと確認）
- PR作成で自動プレビューが見られる仕組みを用意
- 教授は公開前に見た目確認
- あなたはレビューを短時間で実施

（利用サービスは後日決定でOK。まずは『PRで見られる』状態を優先）

## 3-11. 親リポジトリ（okubo-hp）への反映
親リポジトリ https://github.com/rossoandoy/okubo-hp.git にモック（professor-s-keio-portal）の内容を反映する手順。

### 3-11-1. パターン A：親リポジトリにモックを直接 push（推奨）
サイト本体を親リポジトリのルートに置く場合。

1) モック用ディレクトリ（例：`professor-s-keio-portal`）で作業
2) 親リポジトリをリモートに追加（既存の origin はそのまま）
   - `git remote add parent https://github.com/rossoandoy/okubo-hp.git`
3) 親の main に push
   - `git push parent main`
   - 初回で親が空の場合は `git push -u parent main`

※ 今後、ドキュメント（docs/）を親に置くなど運用を変える場合は、親をメタリポジトリにしてサブモジュール構成に変更可能。

### 3-11-2. パターン B：サブモジュールとして親に含める
親リポジトリを「メタリポジトリ」として、モックをサブモジュールで参照する場合。

1) 親リポジトリを clone
2) 親のルートで `git submodule add <モックの clone URL> professor-s-keio-portal`
3) モックの更新は、各リポジトリで通常どおり commit / push したうえで、親で `git submodule update --remote` して親を commit / push
